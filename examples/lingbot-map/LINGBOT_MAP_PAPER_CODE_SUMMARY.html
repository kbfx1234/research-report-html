Technical Contributions

四大创新点

Innovation 01 — Geometric Context Attention
结构化历史管理：GCA 三层上下文

传统 full attention 无法在线处理；普通 causal attention 若保留所有历史 image token 则线性增长；纯 sliding window 会丢失长期轨迹信息。GCA 把历史拆成三种粒度：

① Anchor frames（前 n 帧）：完整 token，固定世界坐标和尺度
② Local window（最近 k 帧）：完整 image tokens，局部相对位姿和稠密深度
③ Trajectory memory（更早历史）：仅保留 6 个 special tokens（camera + 4×register + scale）

复杂度对比：GCA 每帧上下文约 (n+k)×M + 6T，naive causal 为 T×(M+6)。
当 M≈500 时，每个逐出帧仅新增 6 个 token，增长率低约 80×。
示例 n=3, k=16, T=10000：causal ≈ 5×10⁶ tokens，GCA ≈ 7×10⁴ tokens。
Innovation 02 — SLAM-like Memory，端到端学习
把经典 SLAM 状态变成可学习的 Attention 结构

reference frame → anchor（尺度和坐标 grounding）
local map → pose-reference window（局部 dense geometry + relative pose）
global trajectory → trajectory memory（低成本历史 context，抑制漂移）

与传统 SLAM 的核心区别：无手写特征匹配、回环检测或 BA，让 attention 学会在三类上下文间取信息，保留 feed-forward 速度和端到端训练优势。

Innovation 03 — 两阶段渐进训练
Progressive View Training + Relative Pose Loss

Stage 1：VGGT-like offline base model。DINOv2 ViT-L，24层 frame+global attention，随机 2–24 views，建立通用多视图几何先验。

Stage 2：global attention 替换为 GCA，从 Stage 1 权重初始化，视角数 24 → 320，local window 随机采样 16–64，context parallelism 解决显存瓶颈。

额外损失：local window 内所有帧对的 relative pose loss；使用 camera-to-world 监督（区别于 VGGT 的 world-to-camera），降低长序列旋转误差对平移估计的耦合。

Innovation 04 — Runtime Paged KV Cache
FlashInfer 双流 Page 设计，实现实时推理

Patch pages：scale frames 的 patch pages 永久保留；recent window 超窗后回收到 free list。
Special pages：每帧 6 个 special tokens 连续打包，append-only，不回收。
Visible attention pages 顺序：scale patch pages → live window patch pages → all special pages。

论文报告：518×378 输入、1000 帧、window=64 下，FlashInfer ≈ 20 FPS，PyTorch contiguous KV baseline ≈ 10.5 FPS；显存 13.28 GB vs full causal 36.06 GB。

Comparison

相较于 VGGT 的优缺点

优势
  • 真正在线流式：frame-by-frame causal inference，不需要未来帧，贴近 SLAM/VO 任务定义
  • 长序列成本可控：patch page pool 固定大小，special pages 仅 6 tokens/frame，20 FPS @ 1000 帧
  • 更抗长程漂移：Oxford ATE 6.42 vs VGGT 24.78，anchor+window+memory 专为漂移控制设计
  • 更接近工程实时系统：含完整 demo、benchmark、profiling、windowed 对齐 pipeline
  • Stage 1 checkpoint 可复用：可加载进 VGGT 做 bidirectional inference
局限
  • 失去全双向 context：current frame 不能看未来帧，offline 少量 view 全局重建 VGGT 仍更优
  • 输出任务少于 VGGT：无完整 point tracks / dense track head，缺跨视角匹配特征
  • Runtime 更复杂：依赖 FlashInfer、paged cache、keyframe、Sim(3) alignment，调试成本高
  • Direct mode 有训练长度边界：建议 ≤3000 帧，超长需 windowed reset，边界引入 Sim(3) 误差
  • 初始 anchor 帧很关键：开头纹理差或运动退化会拖累全局坐标，VGGT 离线设置更鲁棒
Implementation Notes

代码与论文差异 / 注意事项

NOTE 01
Adaptive Keyframe：论文描述 ≠ 默认 demo 路径
论文 Sec. 4.4 描述基于 optical flow magnitude 的自适应 keyframe。实际上 inference_streaming() 主要走固定 keyframe_interval；flow-based 策略需走 windowed/benchmark 入口（flow_threshold > 0）。
NOTE 02
命名差异：anchor token vs scale_token
论文称 learnable anchor token，代码里是 scale_token，总 special tokens 数为 6。语义上承担 anchor/scale frame 标识和尺度初始化功能。
NOTE 03
训练代码缺失
Repo 中无 Stage 1/Stage 2 完整训练 pipeline。Progressive view training、context parallelism、relative pose loss 的实现细节仅可从论文复述，不能逐行验证。
⚠ POSE
Pose Convention 需复核：C2W / W2C 注释不一致
demo.py 将 pose_encoding_to_extri_intri() 输出再 inverse 一次（注释"Convert w2c to c2w"）；benchmark/methods/lingbot_map.py 则直接写入（注释"C2W"）；utils/pose_enc.py docstring 仍描述为 camera-from-world。接入外部系统前务必用有 GT 的短序列验证方向。
Reproduction Guide

复现与使用建议

01 先跑短序列 demo：使用 example/courthouse 或 example/oxford，确认 checkpoint、pose/depth 输出和可视化链路正常。
02 跑 benchmark adapter：benchmark/configs/oxford.yaml 或 kitti.yaml，重点验证 pose convention 方向是否正确。
03 对超过 320 帧的序列，比较 keyframe_interval=1、auto interval 和 windowed mode 的轨迹稳定性与显存占用。
04 接入机器人系统时，优先检查 C2W/W2C 输出方向、尺度稳定性、sky/dynamic object filtering，以及是否需要外部 loop closure / BA。
05 复用 VGGT 能力：README 中 lingbot-map-stage1 checkpoint 可加载进 VGGT 做 bidirectional inference；Stage 2 才是 streaming GCA 特化。
